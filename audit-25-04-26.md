# Spectre Quant: Rolling Window Approach Audit (Updated)
**Date:** April 25, 2026

*Correction Note: This audit evaluates the actual production pipeline files (`train_rolling_master.py` and `sidecar.py`), correcting a previous misread of an older scratchpad file (`train_rolling_window.py`).*

## 1. Strengths & Solid Architecture
*   **True Inference Parity:** `sidecar.py` properly fetches its own 1-minute data and executes the exact same `build_rolling_5m_windows` logic as training. This ensures the live model distribution perfectly matches the training distribution.
*   **Robust NaN Handling:** During live inference, `sidecar.py` safely wraps every technical indicator computation in a try/except block with neutral fallback values (e.g., RSI=50), followed by an `np.nan_to_num` pass. This makes the inference highly resilient to missing API data.
*   **Nested Cross-Validation:** Using an 80/20 train/test split for a final holdout, combined with `TimeSeriesSplit(n_splits=3)` during hyperparameter tuning, is the correct, rigorous approach for time-series modeling.
*   **Deep Dataset:** Utilizing the 10-year 1-minute dataset provides a massive, highly representative sample size compared to the standard yfinance 7-day limits.

## 2. Identified Vulnerabilities (Now Patched)

### A. Target Leakage via Purge Gap (Fixed)
*   **Issue:** The predictive target looks `forward_bars = 30` ahead. Because the data overlaps at the 1-minute level, the last 30 minutes of `X_train` possessed targets that mathematically bled into the feature-space of the out-of-sample `X_test`.
*   **Fix Applied:** We introduced a manual purge gap, dropping the last 30 rows of `X_train` and `sample_weights` right before model fitting to enforce strict isolation. *(Note: While this fixes the main train/test split, the inner `TimeSeriesSplit` folds still experience minor boundary leakage. For absolute purity in the future, consider implementing a custom `PurgedKFold` splitter).*

### B. Eval Set Leakage Risk (Fixed)
*   **Issue:** The final fit operation included `eval_set=[(X_test, y_test)]`. While not a direct leak unless `early_stopping_rounds` is enabled, passing the true holdout set into the fitting mechanism is considered poor hygiene and creates a severe vulnerability if hyperparameters are ever adjusted.
*   **Fix Applied:** The `eval_set` argument was removed from the final `.fit()` call.

---
**Conclusion:** The live inference path is robust and architecturally sound. The data leakage flaws in the training script have been successfully patched.
